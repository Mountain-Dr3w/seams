# The Artifacts Changed

*Seams — Blog No. 03 | April 2026*

*On what happens when your project artifacts have to work for humans and models at the same time.*

---

The artifacts changed. That's one of the things nobody really talks about when they talk about building with AI.

We still have user stories, design docs, acceptance criteria, ADRs, README files, component specs, coding standards, architecture notes. None of those artifacts disappeared. What changed is that they now have a second audience. They used to be read by teammates. Now they're also read by models that generate code, pick up tasks, and help people recover context they lost while they were away. That sounds like a small shift until you feel the consequences, and then it starts changing how you write everything.

A lot of project artifacts used to work because humans are good at repairing missing context. A ticket could be a little vague because the engineer knew the codebase. An acceptance criterion could be underspecified because the PM and designer had already talked it through. A note like "follow the existing pattern" could stand in for a real explanation because the team shared enough history to know what that meant. None of that was ideal, but it was often good enough.

Models are much worse at that kind of recovery. They're good at producing output from available context and bad at reconstructing context that never made it into the artifact in the first place. If the important constraint only lives in someone's head, in a Slack thread from three weeks ago, or in the fact that the senior engineer on the team "just knows" why a certain decision got made, the model doesn't have it. It will fill the gap with something statistically reasonable, which is not the same as being locally right.

· · ·

We don't need new kinds of artifacts. User stories, component specs, ADRs, coding standards have always been the artifacts that let people with different roles coordinate work without needing to be the same kind of expert. What AI changes is the standard those artifacts have to meet. If the artifact now needs to work for both humans and models on the same project, it has to carry more of the context explicitly. The cost of leaving things implicit went up.

That doesn't mean everything has to become longer. I actually think the opposite is usually better. Long artifacts full of vague prose aren't especially useful to humans or models. What matters is structure and precision. What problem is being solved, what constraints exist, what patterns should be followed, what's already been decided.

In Velveteen, the best example is STANDARDS.md. I built Velveteen to generate a coding standards file for every project based on the actual patterns in the codebase, because I kept watching the same thing happen. I'd describe a feature to Claude, get code back that worked but didn't match the conventions already in the project, and then spend time cleaning up inconsistencies that shouldn't have existed. The model wasn't doing anything wrong. It just didn't know the local rules because I'd never written them down. "Just use the existing style" is not enough when those patterns aren't visible yet.

So STANDARDS.md captures naming, file structure, import conventions, environment variable handling, security basics, and error-handling patterns. Not as process theater. As a way to make the project legible to the next model that touches it, which when you're a designer who's also building, is usually you five minutes from now.

· · ·

User stories and acceptance criteria are where this pressure shows up most in daily work. A thin ticket that says "add export feature" might be enough if the engineer already knows the product and can grab you for ten minutes after standup. It's much less useful when you're using a model to help build it. The story has to carry more of its own weight. What format? From what screen? What permissions? What happens if the dataset is too large? What does the user see while it's running? What does failure look like?

Those aren't "extra details for the AI." They're the details that should have been there all along and often weren't because a human could fill the gaps in real time. The model just makes the cost of that gap visible. And if you're the one writing the story AND the one building from it, you feel both sides of that cost. You know exactly which details you left out because you figured you'd remember, and then you watch the model guess wrong about all of them. ADRs that record *why* a decision exists protect the project from suggestions that sound reasonable but go against choices the team already made, whether those suggestions come from a new hire or a language model. Component specs that name all states (loading, empty, disabled, error, partial success) and not just the happy path are better specs for everyone involved.

· · ·

Teams build memory through artifacts. If all the important context lives in Slack threads and one-off prompts that disappear into someone's conversation history, the project gets brittle. Every new contributor has to rediscover the same rules. Every model call starts from partial amnesia. The work gets done, but it doesn't accumulate understanding very well.

Good artifacts fix some of that. They make decisions stick. They let context survive after the people who made the decision move on or move teams. They reduce the amount of project knowledge that depends on who happened to be in the room when something got decided. I think the teams that benefit most from AI are probably not the ones with the cleverest prompts. They're the ones with the clearest artifacts. A prompt is a local instruction. An artifact is shared memory.

The artifacts changed because the audience changed. The good ones are going to feel less like paperwork and more like interfaces between everyone participating in the work, human or otherwise.
